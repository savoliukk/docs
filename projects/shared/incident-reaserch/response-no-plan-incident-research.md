# Дослідження інцидентів: Немає плану реагування на інциденти

## Контекст з матриці

- Домен: Реагування на інциденти
- Зона: План реагування
- Загроза: Немає плану реагування на інциденти
- Ризик: Без заздалегідь описаного плану команда може втратити час на узгодження дій під час компрометації акаунта, витоку секрету, ransomware, зламу GitHub або витоку хмарного ключа.
- Поточна пріоритетність: Висока

## Пошуковий фокус

Шукались реальні публічні інциденти, де підтверджено або чітко описано схожий розрив у контролях чи ризиковий сценарій для теми: `Немає плану реагування на інциденти`. Пріоритет надавався офіційним післяінцидентним розборам, урядовим попередженням, регуляторним і судовим документам, звітам постачальників безпекових досліджень і надійним медіа. Усі описи нижче подані українською мовою; назви джерел, компаній, продуктів і технічних ідентифікаторів залишені в оригіналі.

- Повторювані патерни: процес сповіщення й реагування не забезпечили своєчасну дію на сигнали безпеки. ; Патчинг, облік власників активів і моніторинг health не були достатньо керованими як процес. ; Секрети зберігались у GitHub, а доступ до GitHub залежав від персональних акаунтів без достатньої MFA-політики..
- Наскільки поширений тип проблеми: знайдені кейси показують, що цей клас розриву трапляється в реальних інцидентах із SaaS-сервісами, хмарними середовищами, CI/CD, пристроями або процесами управління безпекою; точна поширеність не оцінювалась кількісно.
- Чи варто залишити поточну пріоритетність: поточну пріоритетність `Висока` варто залишити, якщо внутрішній стан контролів відповідає описаному ризику; для непрямих кейсів це слід трактувати як підтримувальний, а не абсолютний доказ.

[Equifax - Apache Struts breach and monitoring gaps](response-no-plan-incident-research.md): Equifax 2017 демонструє, що прогалини в обліку власників активів, оновленнях і моніторингу можуть разом приховати масовий злам/витік для ризику `Немає плану реагування на інциденти`. [Target - missed security alerts](response-no-plan-incident-research.md): Target 2013 показує, що навіть наявні сповіщення безпеки не допомагають без відповідальної маршрутизації сповіщень і реагування для ризику `Немає плану реагування на інциденти`. [GitLab.com - database incident and failed backups](response-no-plan-incident-research.md): GitLab 2017 показує, що резервне копіювання без перевіреного відновлення може не спрацювати саме під час інциденту для ризику `Немає плану реагування на інциденти`. [Maersk - NotPetya recovery](response-no-plan-incident-research.md): Maersk NotPetya показує, що без ізольованого DR і аварійної координації відновлення може стати ручним і хаотичним для ризику `Немає плану реагування на інциденти`.
