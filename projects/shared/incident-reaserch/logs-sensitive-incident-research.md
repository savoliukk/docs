# Дослідження інцидентів: Журнали як сховище чутливих даних

## Контекст з матриці

- Домен: Журнали, моніторинг і виявлення атак
- Зона: Системні та прикладні журнали
- Загроза: Журнали як сховище чутливих даних
- Ризик: Журнали можуть містити IP, ідентифікатори користувачів, сліди корисного навантаження запитів, токени, помилки, версії бібліотек і внутрішню архітектуру. Зловмисник може використати це для точніших атак або пошуку секретів.
- Поточна пріоритетність: Середня

## Пошуковий фокус

Шукались реальні публічні інциденти, де підтверджено або чітко описано схожий розрив у контролях чи ризиковий сценарій для теми: `Журнали як сховище чутливих даних`. Пріоритет надавався офіційним післяінцидентним розборам, урядовим попередженням, регуляторним і судовим документам, звітам постачальників безпекових досліджень і надійним медіа. Усі описи нижче подані українською мовою; назви джерел, компаній, продуктів і технічних ідентифікаторів залишені в оригіналі.

## Інциденти

### 1. Twitter - plaintext passwords in internal log

- Рік і дата інциденту або публікації: травень 2018
- Тип джерела: повідомлення компанії / reliable media
- Джерела:
  - [CNBC coverage of Twitter disclosure](https://www.cnbc.com/2018/05/03/twitter-says-it-fixed-a-password-problem-but-that-you-should-still-change-yours-anwway.html)
- Короткий опис: Twitter повідомила користувачам про помилку, через яку паролі записувалися до внутрішнього журналу перед хешування.
- Використаний розрив у контролях або ризиковий розрив: Application logging випадково зберігав секрети до їхнього захисту.
- Потенційний або підтверджений подальший рух атакувальника: Компанія не повідомляла про зловживання, але внутрішнього журналу із паролі у відкритому вигляді створює високий ризик у разі доступу до журналів.
- Наслідки: Twitter рекомендувала користувачам змінити паролі.
- Релевантність до цієї загрози/ризику: Релевантність пряма: кейс підтверджує або ілюструє розрив у контролях або ризиковий розрив для загрози 'Журнали як сховище чутливих даних' у зоні 'Системні та прикладні журнали', бо показує реальний наслідок схожого розриву: Application logging випадково зберігав секрети до їхнього захисту.
- Рівень доказовості: середній
- Чи підсилює пріоритетність ризику: так
- Згадки про інструменти безпеки, якщо є: application журнали, password хешування

### 2. Facebook - passwords stored in readable internal systems

- Рік і дата інциденту або публікації: березень 2019
- Тип джерела: official company blog
- Джерела:
  - [Meta/Facebook, Keeping Passwords Secure](https://about.fb.com/news/2019/03/keeping-passwords-secure.)

- Використаний розрив у контролях або ризиковий розрив: Внутрішні storage/logging процеси містили секрети у читабельному вигляді.
- Потенційний або підтверджений подальший рух атакувальника: Компанія не заявила про зловживання, але читабельне внутрішнє сховище збільшує ризик інсайдерського сценарію або подальшої компрометації.
- Наслідки: Facebook повідомляла сотні мільйонів Facebook Lite користувачів, десятки мільйонів інших користувачів і Instagram користувачів.
- Релевантність до цієї загрози/ризику: Релевантність пряма: кейс підтверджує або ілюструє розрив у контролях або ризиковий розрив для загрози 'Журнали як сховище чутливих даних' у зоні 'Системні та прикладні журнали', бо показує реальний наслідок схожого розриву: Внутрішні storage/logging процеси містили секрети у читабельному вигляді.
- Рівень доказовості: високий
- Чи підсилює пріоритетність ризику: так
- Згадки про інструменти безпеки, якщо є: внутрішнє сховище, робота з паролями

- Релевантність до цієї загрози/ризику: Релевантність пряма: кейс підтверджує або ілюструє розрив у контролях або ризиковий розрив для загрози 'Журнали як сховище чутливих даних' у зоні 'Системні та прикладні журнали', бо показує реальний наслідок схожого розриву: Administrative password recovery/control flow створив довготривалий secret-handling розрив.

## Висновок

- Повторювані патерни: Administrative password recovery/control flow створив довготривалий secret-handling розрив. ; Application logging випадково зберігав секрети до їхнього захисту. ; Внутрішні storage/logging процеси містили секрети у читабельному вигляді..
- Наскільки поширений тип проблеми: знайдені кейси показують, що цей клас розриву трапляється в реальних інцидентах із SaaS-сервісами, хмарними середовищами, CI/CD, пристроями або процесами управління безпекою; точна поширеність не оцінювалась кількісно.
- Чи варто залишити поточну пріоритетність: поточну пріоритетність `Середня` варто залишити, якщо внутрішній стан контролів відповідає описаному ризику; для непрямих кейсів це слід трактувати як підтримувальний, а не абсолютний доказ.
- Короткий текст для колонки "Пов'язані інциденти": [Twitter - plaintext passwords in internal log](logs-sensitive-incident-research.md): Twitter 2018 показує, як application logging може випадково зберегти паролі у відкритому вигляді для ризику `Журнали як сховище чутливих даних`. [Facebook - passwords stored in readable internal systems](logs-sensitive-incident-research.md): Facebook 2019 показує, що внутрішні журнали/сховище можуть перетворитися на сховище паролів для ризику `Журнали як сховище чутливих даних`. [Google G Suite - unhashed passwords](logs-sensitive-incident-research.md): Google G Suite 2019 показує, що admin control flow може роками зберігати passwords у небезпечному форматі для ризику `Журнали як сховище чутливих даних`.
