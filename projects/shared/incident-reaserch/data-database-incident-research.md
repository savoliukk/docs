# Дослідження інцидентів: Прямий доступ до критичних даних

## Контекст з матриці

- Домен: Дані, бази даних і сховища файлів
- Зона: База даних
- Загроза: Прямий доступ до критичних даних
- Ризик: У тезах база даних описана як сховище криптографічних ключів пристроїв, персональних даних і журналів подій. Зловмисник може викрасти дані, змінити журнали або підробляти команди/трафік пристроїв.
- Поточна пріоритетність: Висока

## Пошуковий фокус

Шукались реальні публічні інциденти, де підтверджено або чітко описано схожий розрив у контролях чи ризиковий сценарій для теми: `Прямий доступ до критичних даних`. Пріоритет надавався офіційним післяінцидентним розборам, урядовим попередженням, регуляторним і судовим документам, звітам постачальників безпекових досліджень і надійним медіа. Усі описи нижче подані українською мовою; назви джерел, компаній, продуктів і технічних ідентифікаторів залишені в оригіналі.

### 1. Capital One - cloud data breach

- Статус дедуплікації: повний первинний опис інциденту подано в [k8s-rbac](k8s-rbac-incident-research.md); тут наведено тільки стислий зв'язок із поточним ризиком.
- Рік і дата інциденту або публікації: 2019
- Тип джерела: government / court record
- Джерела:
  - [DOJ conviction announcement](https://www.justice.gov/usao-wdwa/pr/seattle-tech-worker-convicted-wire-fraud-and-computer-intrusions-capital-one-data)
- Стислий опис: Capital One 2019 показує, як розрив у Cloud IAM може відкрити доступ до масиву критичних даних.
- Використаний розрив у контролях або ризиковий розрив у цьому рядку: Cloud IAM/metadata/service boundary дозволив отримати доступ до великих обсягів даних.
- Наслідки для цього контексту: Capital One disclosed data breach для понад 100 млн заявок/клієнтів; інцидент мав регуляторні та судові наслідки.
- Релевантність до цієї загрози/ризику: Релевантність пряма: кейс підтверджує або ілюструє розрив у контролях або ризиковий розрив для загрози 'Прямий доступ до критичних даних' у зоні 'База даних', бо показує реальний наслідок схожого розриву: Cloud IAM/metadata/service boundary дозволив отримати доступ до великих обсягів даних.
- Рівень доказовості: високий
- Чи підсилює пріоритетність ризику: так
- Згадки про інструменти безпеки, якщо є: AWS, IAM role, S3

### 3. Block / Cash App Investing - доступ колишнього співробітника

- Статус дедуплікації: повний первинний опис інциденту подано в [access-offboarding](access-offboarding-incident-research.md); тут наведено тільки стислий зв'язок із поточним ризиком.

- Тип джерела: regulatory filing

- Рівень доказовості: високий
- Чи підсилює пріоритетність ризику: так
- Згадки про інструменти безпеки, якщо є: звітність Cash App Investing

- Статус дедуплікації: повний первинний опис інциденту подано в [access-idp-source](access-idp-source-incident-research.md); тут наведено тільки стислий зв'язок із поточним ризиком.

- Джерела:
  - [Heroku April 2022 Incident Review](https://www.heroku.com/blog/april-2022-incident-review/)

- Наслідки для цього контексту: Heroku відкликала токени, повідомляла клієнтів і ротувала доступи; потенційно постраждали приватні репозиторії й секрети.
- Релевантність до цієї загрози/ризику: Релевантність пряма: кейс підтверджує або ілюструє розрив у контролях або ризиковий розрив для загрози 'Прямий доступ до критичних даних' у зоні 'База даних', бо показує реальний наслідок схожого розриву: Збережені OAuth-токени й конфігураційні змінні конвеєра доставки створили широкий радіус ураження між платформою як сервісом (PaaS), GitHub і CI/CD.
- Рівень доказовості: високий
- Чи підсилює пріоритетність ризику: так

- Релевантність до цієї загрози/ризику: Релевантність пряма: кейс ілюструє актуальний прояв такого розриву у 2025/2026 для загрози 'Прямий доступ до критичних даних' у зоні 'База даних', оскільки Довірена SaaS/AI-інтеграція з довгоживучими OAuth-токени дала міжклієнтський радіус ураження для клієнтські CRM-дані.

## Висновок

- Повторювані патерни: Cloud IAM/metadata/service boundary дозволив отримати доступ до великих обсягів даних. ; Offboarding не прибрав або не заблокував доступ до звітів, які були потрібні співробітнику в минулій ролі. ; Внутрішня база support analytics була помилково доступна з інтернету..
- Наскільки поширений тип проблеми: знайдені кейси показують, що цей клас розриву трапляється в реальних інцидентах із SaaS-сервісами, хмарними середовищами, CI/CD, пристроями або процесами управління безпекою; точна поширеність не оцінювалась кількісно.
- Чи варто залишити поточну пріоритетність: поточну пріоритетність `Висока` варто залишити, якщо внутрішній стан контролів відповідає описаному ризику; для непрямих кейсів це слід трактувати як підтримувальний, а не абсолютний доказ.
- Короткий текст для колонки "Пов'язані інциденти": [Capital One - cloud data breach](data-database-incident-research.md): Capital One 2019 показує, як розрив у Cloud IAM може відкрити доступ до масиву критичних даних для ризику `Прямий доступ до критичних даних`. [Microsoft - exposed customer support database](data-database-incident-research.md): Microsoft 2020 показує, що помилково налаштована база технічної підтримки може розкрити клієнтський контекст без окремої атаки на застосунок для ризику `Прямий доступ до критичних даних`. [Block / Cash App Investing - доступ колишнього співробітника](data-database-incident-research.md): Cash App Investing показує прямий ризик розрив у процесі відкликання доступів після звільнення, коли колишній співробітник зберіг доступ до клієнтських звітів для ризику `Прямий доступ до критичних даних`. [Shopify - rogue support employees](data-database-incident-research.md): Shopify 2020 підкреслює ризик надмірного доступу технічної підтримки й слабкої персональної відповідальності для ризику `Прямий доступ до критичних даних`. [Heroku - stolen GitHub OAuth tokens and config vars](data-database-incident-research.md): Heroku 2022 показує, що OAuth-токени та конфігураційні секрети можуть зв'язати платформу як сервіс (PaaS), GitHub і CI/CD в один шлях компрометації для ризику `Прямий доступ до критичних даних`.
