# Дослідження інцидентів: Секрети були або є в репозиторіях

## Контекст з матриці

- Домен: Паролі, секрети та криптографічні ключі
- Зона: Секрети виконання
- Загроза: Секрети були або є в репозиторіях
- Ризик: Якщо секрет потрапляє в Git, він може залишитись в історії або копіях репозиторію. Це відкриває шлях до баз даних, хмари, CI/CD або Kubernetes навіть після видалення з поточної версії файлу.
- Поточна пріоритетність: Висока

## Пошуковий фокус

Шукались реальні публічні інциденти, де підтверджено або чітко описано схожий розрив у контролях чи ризиковий сценарій для теми: `Секрети були або є в репозиторіях`. Пріоритет надавався офіційним післяінцидентним розборам, урядовим попередженням, регуляторним і судовим документам, звітам постачальників безпекових досліджень і надійним медіа. Усі описи нижче подані українською мовою; назви джерел, компаній, продуктів і технічних ідентифікаторів залишені в оригіналі.

### 1. Uber 2016 - GitHub credentials and AWS S3 data

- Статус дедуплікації: повний первинний опис інциденту подано в [access-mfa](access-mfa-incident-research.md); тут наведено тільки стислий зв'язок із поточним ризиком.
- Рік і дата інциденту або публікації: жовтень-листопад 2016; FTC complaint 2018
- Тип джерела: regulatory complaint
- Джерела:
  - [FTC revised complaint](https://www.ftc.gov/system/files/documents/cases/1523054_uber_technologies_revised_complaint_0.pdf)
- Стислий опис: Uber 2016 є прямим прикладом, як секрет у GitHub відкрив шлях до AWS S3 datastore.
- Використаний розрив у контролях або ризиковий розрив у цьому рядку: Секрети зберігались у GitHub, а доступ до GitHub залежав від персональних акаунтів без достатньої MFA-політики.
- Наслідки для цього контексту: Інцидент зачепив десятки мільйонів користувачів і водіїв та призвів до регуляторних наслідків.
- Релевантність до цієї загрози/ризику: Релевантність пряма: кейс підтверджує або ілюструє розрив у контролях або ризиковий розрив для загрози 'Секрети були або є в репозиторіях' у зоні 'Секрети виконання', бо показує реальний наслідок схожого розриву: Секрети зберігались у GitHub, а доступ до GitHub залежав від персональних акаунтів без достатньої MFA-політики.
- Рівень доказовості: високий
- Чи підсилює пріоритетність ризику: так

### 4. Heroku - stolen GitHub OAuth tokens and config vars

- Статус дедуплікації: повний первинний опис інциденту подано в [access-idp-source](access-idp-source-incident-research.md); тут наведено тільки стислий зв'язок із поточним ризиком.
- Рік і дата інциденту або публікації: квітень-травень 2022
- Тип джерела: офіційний розбір інциденту
- Джерела:
  - [Heroku April 2022 Incident Review](https://www.heroku.com/blog/april-2022-incident-review/)
- Стислий опис: Heroku 2022 показує, що OAuth-токени та конфігураційні секрети можуть зв'язати платформу як сервіс (PaaS), GitHub і CI/CD в один шлях компрометації.
- Використаний розрив у контролях або ризиковий розрив у цьому рядку: Збережені OAuth-токени й конфігураційні змінні конвеєра доставки створили широкий радіус ураження між платформою як сервісом (PaaS), GitHub і CI/CD.
- Наслідки для цього контексту: Heroku відкликала токени, повідомляла клієнтів і ротувала доступи; потенційно постраждали приватні репозиторії й секрети.
- Релевантність до цієї загрози/ризику: Релевантність пряма: кейс підтверджує або ілюструє розрив у контролях або ризиковий розрив для загрози 'Секрети були або є в репозиторіях' у зоні 'Секрети виконання', бо показує реальний наслідок схожого розриву: Збережені OAuth-токени й конфігураційні змінні конвеєра доставки створили широкий радіус ураження між платформою як сервісом (PaaS), GitHub і CI/CD.
- Рівень доказовості: високий
- Чи підсилює пріоритетність ризику: так

- Дата атаки або виявлення, якщо відома: GitHub виявив і стримав компрометацію 18 травня 2026
- Тип джерела: офіційний блог безпеки / reliable media
- Джерела:
  - [GitHub Blog](https://github.blog/security/investigating-unauthorized-access-to-githubs-internal-repositories/)
  - [TechCrunch](https://techcrunch.com/2026/05/20/github-says-hackers-stole-data-from-thousands-of-internal-repositories/)
- Стислий опис українською: GitHub 2026 показує, що шкідливий інструмент розробника на одному пристрої (endpoint) може відкрити внутрішні репозиторії і запустити ротацію критичних секретів.
- Використаний розрив у контролях або ризиковий розрив: Компрометований пристрій розробника (endpoint) з доступом до репозиторії з вихідним кодом і секрети створив шлях до внутрішнього коду та облікові дані та пов'язані секрети.

- Релевантність до цієї загрози/ризику: Релевантність пряма: кейс ілюструє актуальний прояв такого розриву у 2025/2026 для загрози 'Секрети були або є в репозиторіях' у зоні 'Секрети виконання', оскільки Компрометований пристрій розробника (endpoint) з доступом до репозиторії з вихідним кодом і секрети створив шлях до внутрішнього коду та облікові дані та пов'язані секрети.

- Наскільки поширений тип проблеми: знайдені кейси показують, що цей клас розриву трапляється в реальних інцидентах із SaaS-сервісами, хмарними середовищами, CI/CD, пристроями або процесами управління безпекою; точна поширеність не оцінювалась кількісно.
- Чи варто залишити поточну пріоритетність: поточну пріоритетність `Висока` варто залишити, якщо внутрішній стан контролів відповідає описаному ризику; для непрямих кейсів це слід трактувати як підтримувальний, а не абсолютний доказ.
- Короткий текст для колонки "Пов'язані інциденти": [Uber 2016 - GitHub credentials and AWS S3 data](secrets-git-incident-research.md): Uber 2016 є прямим прикладом, як секрет у GitHub відкрив шлях до AWS S3 datastore для ризику `Секрети були або є в репозиторіях`. [Toyota T-Connect - access key exposed on GitHub](secrets-git-incident-research.md): Toyota T-Connect показує, що ключ доступу у GitHub може роками створювати шлях до дані клієнтів для ризику `Секрети були або є в репозиторіях`. [Microsoft AI researchers - exposed Azure SAS token](secrets-git-incident-research.md): Microsoft AI SAS 2023 показує, що один надмірний хмарний токен спільного доступу може розкрити великий обсяг внутрішні дані для ризику `Секрети були або є в репозиторіях`. [Heroku - stolen GitHub OAuth tokens and config vars](secrets-git-incident-research.md): Heroku 2022 показує, що OAuth-токени та конфігураційні секрети можуть зв'язати платформу як сервіс (PaaS), GitHub і CI/CD в один шлях компрометації для ризику `Секрети були або є в репозиторіях`. [Twitch - source code and payout data leak](secrets-git-incident-research.md): Twitch 2021 показує, що витік репозиторію може розкрити вихідний код і внутрішній бізнес-контекст для ризику `Секрети були або є в репозиторіях`.
