# Дослідження інцидентів: Ручні зміни в бойовому середовищі через веб-консоль

## Контекст з матриці

- Домен: Бойове середовище і контрольні площини
- Зона: Сервер і процеси
- Загроза: Ручні зміни в бойовому середовищі через веб-консоль
- Ризик: Ручні зміни можуть обійти контрольовану доставку змін і ускладнити аудит. У разі помилки або компрометації акаунта зміни можуть вплинути на бойове середовище без очікуваної перевірки.
- Поточна пріоритетність: Середня

## Пошуковий фокус

Шукались реальні публічні інциденти, де підтверджено або чітко описано схожий розрив у контролях чи ризиковий сценарій для теми: `Ручні зміни в бойовому середовищі через веб-консоль`. Пріоритет надавався офіційним післяінцидентним розборам, урядовим попередженням, регуляторним і судовим документам, звітам постачальників безпекових досліджень і надійним медіа. Усі описи нижче подані українською мовою; назви джерел, компаній, продуктів і технічних ідентифікаторів залишені в оригіналі.

## Інциденти

### 1. GitLab.com - database incident and failed backups

- Рік і дата інциденту або публікації: січень-лютий 2017
- Тип джерела: офіційний післяінцидентний розбір
- Джерела:
  - [GitLab database incident](https://about.gitlab.com/blog/gitlab-dot-com-database-incident/)
  - [GitLab postmortem](https://about.gitlab.com/blog/postmortem-of-database-outage-of-january-31/)
- Короткий опис: GitLab описав інцидент із видаленням бойове середовище database data та виявленням, що кілька резервне копіювання mechanisms не дали очікуваного відновлення.
- Використаний розрив у контролях або ризиковий розрив: Завдання резервного копіювання і збої моніторингу не гарантували відновлення в момент інциденту.
- Потенційний або підтверджений подальший рух атакувальника: Це не була атака, але operational failure показав той самий risk розрив для відновлення readiness.
- Наслідки: GitLab відновлювався з обмеженого snapshot і втратив частину даних за визначений період.
- Релевантність до цієї загрози/ризику: Релевантність пряма: кейс підтверджує або ілюструє розрив у контролях або ризиковий розрив для загрози 'Ручні зміни в бойовому середовищі через веб-консоль' у зоні 'Сервер і процеси', бо показує реальний наслідок схожого розриву: Завдання резервного копіювання і збої моніторингу не гарантували відновлення в момент інциденту.
- Рівень доказовості: високий
- Чи підсилює пріоритетність ризику: так
- Згадки про інструменти безпеки, якщо є: PostgreSQL, резервні копії, моніторинг

### 3. PHP - compromise of git.php.net

- Рік і дата інциденту або публікації: березень 2021
- Тип джерела: project публічне розкриття / reliable technical media
- Джерела:
  - [PHP.Watch incident update](https://php.watch/news/2021/04/php-src-attack)
  - [Ars Technica coverage](https://arstechnica.com/gadgets/2021/03/hackers-backdoor-php-source-code-after-breaching-internal-git-server/)
- Короткий опис: Офіційний Git server PHP отримав два шкідливі зміни в коді, що імітували авторство відомих maintainer-ів; команда відмовилась від git.php.net і перейшла на GitHub.
- Використаний розрив у контролях або ризиковий розрив: Self-hosted сервер контролю версій мав достатній рівень довіри, щоб шкідливий коміт виглядав легітимним.
- Потенційний або підтверджений подальший рух атакувальника: Якби шкідливий коміт потрапив у реліз, він міг би вплинути на downstream користувачів PHP.
- Наслідки: Код було швидко помічено й вилучено, але проєкт змінив модель source-control.
- Релевантність до цієї загрози/ризику: Релевантність пряма: кейс підтверджує або ілюструє розрив у контролях або ризиковий розрив для загрози 'Ручні зміни в бойовому середовищі через веб-консоль' у зоні 'Сервер і процеси', бо показує реальний наслідок схожого розриву: Self-hosted сервер контролю версій мав достатній рівень довіри, щоб шкідливий коміт виглядав легітимним.
- Рівень доказовості: середній
- Чи підсилює пріоритетність ризику: так
- Згадки про інструменти безпеки, якщо є: git.php.net, GitHub
