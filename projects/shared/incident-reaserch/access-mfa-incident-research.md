# Дослідження інцидентів: MFA не ввімкнена для всіх користувачів

## Контекст з матриці

- Домен: Керування доступами та обліковими записами
- Зона: IdP / SSO / MFA
- Загроза: MFA не ввімкнена для всіх користувачів
- Ризик: Компрометація пароля може одразу дати доступ до робочих сервісів. Це особливо небезпечно для GitHub, хмари, адмін-панелей і сервісів з доступом до бойового середовища.
- Поточна пріоритетність: Висока

## Пошуковий фокус

Шукались реальні публічні інциденти, де підтверджено або чітко описано схожий розрив у контролях чи ризиковий сценарій для теми: `MFA не ввімкнена для всіх користувачів`. Пріоритет надавався офіційним післяінцидентним розборам, урядовим попередженням, регуляторним і судовим документам, звітам постачальників безпекових досліджень і надійним медіа. Усі описи нижче подані українською мовою; назви джерел, компаній, продуктів і технічних ідентифікаторів залишені в оригіналі.

## Інциденти

### 1. Uber - компрометація через MFA fatigue

- Рік і дата інциденту або публікації: вересень 2022
- Тип джерела: офіційний блог безпеки
- Джерела:
  - [Uber Security Update](https://www.uber.com/newsroom/security-update/)
- Короткий опис: Uber повідомила, що атакувальник отримав облікові дані зовнішнього підрядника, неодноразово ініціював MFA-запити й після прийняття одного з них отримав доступ до внутрішніх систем.
- Використаний розрив у контролях або ризиковий розрив: MFA була наявна, але процес не захищав від push-fatigue та компрометації людського фактора.
- Потенційний або підтверджений подальший рух атакувальника: Після входу атакувальник отримав доступ до кількох внутрішніх систем, що показує ризик бічний рух атакувальника після компрометації одного користувача.
- Наслідки: Uber тимчасово обмежувала доступи й розслідувала вторгнення; публічний опис підкреслює операційний вплив компрометації корпоративного акаунта.
- Релевантність до цієї загрози/ризику: Релевантність пряма: кейс підтверджує або ілюструє розрив у контролях або ризиковий розрив для загрози 'MFA не ввімкнена для всіх користувачів' у зоні 'IdP / SSO / MFA', бо показує реальний наслідок схожого розриву: MFA була наявна, але процес не захищав від push-fatigue та компрометації людського фактора.
- Рівень доказовості: високий
- Чи підсилює пріоритетність ризику: так
- Згадки про інструменти безпеки, якщо є: MFA, корпоративні внутрішні системи

- Наслідки: Компанія зупинила трубопровідні операції, що спричинило паливний дефіцит і значний суспільний резонанс.
- Релевантність до цієї загрози/ризику: Релевантність пряма: кейс підтверджує або ілюструє розрив у контролях або ризиковий розрив для загрози 'MFA не ввімкнена для всіх користувачів' у зоні 'IdP / SSO / MFA', бо показує реальний наслідок схожого розриву: VPN-доступ до корпоративного контуру не був достатньо захищений MFA та життєвим циклом акаунтів.
- Рівень доказовості: середній

### 1. GitHub - витік internal repositories через poisoned VS Code extension

- Дата публічного розкриття, публікації або оновлення: 20-26 травня 2026
- Дата атаки або виявлення, якщо відома: GitHub виявив і стримав компрометацію 18 травня 2026
- Тип джерела: офіційний блог безпеки / reliable media
- Джерела:
  - [GitHub Blog](https://github.blog/security/investigating-unauthorized-access-to-githubs-internal-repositories/)
  - [TechCrunch](https://techcrunch.com/2026/05/20/github-says-hackers-stole-data-from-thousands-of-internal-repositories/)
- Опис інциденту українською: GitHub повідомив про компрометацію пристрою співробітника через шкідливе розширення VS Code і несанкціонований доступ до внутрішніх репозиторіїв; компанія ізолювала endpoint, видалила шкідливу версію extension і почала ротацію критичних секретів.
- Використаний розрив у контролях або ризиковий розрив: Компрометований пристрій розробника (endpoint) з доступом до репозиторії з вихідним кодом і секрети створив шлях до внутрішнього коду та облікові дані та пов'язані секрети.
- Потенційний або підтверджений подальший рух атакувальника: Атакувальник перейшов від developer tooling/endpoint до внутрішні репозиторії; GitHub повідомив, що claims про приблизно 3,800 repositories узгоджуються з розслідуванням.
- Наслідки: За публічним описом, вплив обмежувався внутрішні репозиторії; GitHub не мав доказів впливу на клієнтські репозиторії outside внутрішні репозиторії, але виконував ротацію секретів і аналіз журналів.
- Релевантність до цієї загрози/ризику: Релевантність пряма: кейс ілюструє актуальний прояв такого розриву у 2025/2026 для загрози 'MFA не ввімкнена для всіх користувачів' у зоні 'IdP / SSO / MFA', оскільки Компрометований пристрій розробника (endpoint) з доступом до репозиторії з вихідним кодом і секрети створив шлях до внутрішнього коду та облікові дані та пов'язані секрети.
- Рівень доказовості: високий
- Чи підсилює пріоритетність ризику: так

## Висновок

- Повторювані патерни: MFA була наявна, але процес не захищав від push-fatigue та компрометації людського фактора. ; VPN-доступ до корпоративного контуру не був достатньо захищений MFA та життєвим циклом акаунтів. ; Людські облікові записи та support-процеси стали шляхом до даних клієнтів..
- Наскільки поширений тип проблеми: знайдені кейси показують, що цей клас розриву трапляється в реальних інцидентах із SaaS-сервісами, хмарними середовищами, CI/CD, пристроями або процесами управління безпекою; точна поширеність не оцінювалась кількісно.
- Чи варто залишити поточну пріоритетність: поточну пріоритетність `Висока` варто залишити, якщо внутрішній стан контролів відповідає описаному ризику; для непрямих кейсів це слід трактувати як підтримувальний, а не абсолютний доказ.
