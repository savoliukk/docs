# Дослідження інцидентів: Немає реєстру ризиків

## Контекст з матриці

- Домен: Управління безпекою і робочі процеси
- Зона: Політики та перевірки
- Загроза: Немає реєстру ризиків
- Ризик: Без реєстру ризиків команда не має єдиного місця для пріоритетів, стану роботи, прийнятих ризиків і доказів перевірки. Це ускладнює послідовне закриття прогалин.
- Поточна пріоритетність: Середня

## Пошуковий фокус

Шукались реальні публічні інциденти, де підтверджено або чітко описано схожий розрив у контролях чи ризиковий сценарій для теми: `Немає реєстру ризиків`. Пріоритет надавався офіційним післяінцидентним розборам, урядовим попередженням, регуляторним і судовим документам, звітам постачальників безпекових досліджень і надійним медіа. Усі описи нижче подані українською мовою; назви джерел, компаній, продуктів і технічних ідентифікаторів залишені в оригіналі.

  - [House Oversight Equifax report](https://oversight.house.gov/wp-content/uploads/2018/12/Equifax-Report.pdf)
  - [House release summary](https://oversight.house.gov/report/committee-releases-report-revealing-new-information-on-equifax-data-breach/)
- Стислий опис: Equifax 2017 демонструє, що прогалини в обліку власників активів, оновленнях і моніторингу можуть разом приховати масовий злам/витік.

- Повторювані патерни: процес сповіщення й реагування не забезпечили своєчасну дію на сигнали безпеки. ; Площина резервного копіювання/керування був доступний з того самого привілейованого хмарного акаунта, що й виробничих артефактів. ; Патчинг, облік власників активів і моніторинг health не були достатньо керованими як процес..
- Наскільки поширений тип проблеми: знайдені кейси показують, що цей клас розриву трапляється в реальних інцидентах із SaaS-сервісами, хмарними середовищами, CI/CD, пристроями або процесами управління безпекою; точна поширеність не оцінювалась кількісно.
- Чи варто залишити поточну пріоритетність: поточну пріоритетність `Середня` варто залишити, якщо внутрішній стан контролів відповідає описаному ризику; для непрямих кейсів це слід трактувати як підтримувальний, а не абсолютний доказ.
- Короткий текст для колонки "Пов'язані інциденти": [Equifax - Apache Struts breach and monitoring gaps](gov-no-risk-register-incident-research.md): Equifax 2017 демонструє, що прогалини в обліку власників активів, оновленнях і моніторингу можуть разом приховати масовий злам/витік для ризику `Немає реєстру ризиків`. [Target - missed security alerts](gov-no-risk-register-incident-research.md): Target 2013 показує, що навіть наявні сповіщення безпеки не допомагають без відповідальної маршрутизації сповіщень і реагування для ризику `Немає реєстру ризиків`. [Code Spaces - AWS console compromise and backup deletion](gov-no-risk-register-incident-research.md): Code Spaces 2014 показує, що резервні копії без ізоляції від площини керування можуть бути знищені разом із бойовим середовищем для ризику `Немає реєстру ризиків`. [GitLab.com - database incident and failed backups](gov-no-risk-register-incident-research.md): GitLab 2017 показує, що резервне копіювання без перевіреного відновлення може не спрацювати саме під час інциденту для ризику `Немає реєстру ризиків`. [OVHcloud Strasbourg - data center fire and backup locality](gov-no-risk-register-incident-research.md): OVHcloud Strasbourg показує, що резервне копіювання у тому самому домені відмови може не пережити фізичний інцидент для ризику `Немає реєстру ризиків`.
