# Дослідження інцидентів: Немає єдиного джерела облікових записів

## Контекст з матриці

- Домен: Керування доступами та обліковими записами
- Зона: IdP / SSO / MFA
- Загроза: Немає єдиного джерела облікових записів
- Ризик: Доступи видаються вручну і не прив'язані до одного життєвого циклу користувача. Зловмисник або колишній співробітник може зберегти доступ до окремого сервісу, якщо його не відкликали вручну.
- Поточна пріоритетність: Висока

## Пошуковий фокус

Шукались реальні публічні інциденти, де підтверджено або чітко описано схожий розрив у контролях чи ризиковий сценарій для теми: `Немає єдиного джерела облікових записів`. Пріоритет надавався офіційним післяінцидентним розборам, урядовим попередженням, регуляторним і судовим документам, звітам постачальників безпекових досліджень і надійним медіа. Усі описи нижче подані українською мовою; назви джерел, компаній, продуктів і технічних ідентифікаторів залишені в оригіналі.

## Інциденти

### 1. Okta / Sitel - компрометація support-доступу Lapsus$

- Рік і дата інциденту або публікації: березень 2022
- Тип джерела: офіційний блог безпеки
- Джерела:
  - [Okta Security, updated statement on Lapsus$](https://sec.okta.com/articles/2022/03/updated-okta-statement-lapsus/)
- Короткий опис: Okta повідомила про інцидент, пов'язаний із стороннім провайдером технічної підтримки Sitel/SYKES; зловмисники отримали доступ до робочого середовища інженера технічної підтримки й могли виконувати обмежені дії в контексті технічної підтримки.
- Використаний розрив у контролях або ризиковий розрив: Доступ зони технічної підтримки/IdP залежав від окремого привілейованого робочого контуру та стороннього провайдера.
- Потенційний або підтверджений подальший рух атакувальника: За даними Okta, доступ був обмежений діями технічної підтримки, але сам клас інциденту показує, як компрометація облікового запису технічної підтримки може стати точкою входу до клієнтського контексту.
- Наслідки: Okta оцінила вплив як обмежений, але інцидент створив репутаційний та операційний ризик для клієнтів, які покладаються на IdP як на центральну точку довіри.
- Релевантність до цієї загрози/ризику: Релевантність пряма: кейс підтверджує або ілюструє розрив у контролях або ризиковий розрив для загрози 'Немає єдиного джерела облікових записів' у зоні 'IdP / SSO / MFA', бо показує реальний наслідок схожого розриву: Доступ зони технічної підтримки/IdP залежав від окремого привілейованого робочого контуру та стороннього провайдера.
- Рівень доказовості: високий
- Чи підсилює пріоритетність ризику: так
- Згадки про інструменти безпеки, якщо є: Okta, інструменти технічної підтримки

### 2. Okta - unauthorized access to support case management system

- Рік і дата інциденту або публікації: вересень-жовтень 2023
- Тип джерела: офіційний звіт про першопричину
- Джерела:
  - [Okta root cause and remediation](https://sec.okta.com/articles/2023/11/unauthorized-access-oktas-support-case-management-system-root-cause/)
- Короткий опис: Okta підтвердила несанкціонований доступ до файлів у системі підтримки клієнтів; першопричина включала облікові дані сервісного акаунта, збережені в особистому Google-акаунті співробітника.
- Використаний розрив у контролях або ризиковий розрив: Файли й артефакти технічної підтримки, HAR-файли й службові облікові дані не були достатньо ізольовані від персонального пристрою/акаунта.
- Потенційний або підтверджений подальший рух атакувальника: HAR-файли могли містити сесійні токени, які Okta відкликала для постраждалих клієнтів.
- Наслідки: Спершу Okta повідомила про 134 клієнтів; пізніше обсяг комунікацій розширювався для користувачів системи підтримки клієнтів.
- Релевантність до цієї загрози/ризику: Релевантність пряма: кейс підтверджує або ілюструє розрив у контролях або ризиковий розрив для загрози 'Немає єдиного джерела облікових записів' у зоні 'IdP / SSO / MFA', бо показує реальний наслідок схожого розриву: Файли й артефакти технічної підтримки, HAR-файли й службові облікові дані не були достатньо ізольовані від персонального пристрою/акаунта.
- Рівень доказовості: високий
- Чи підсилює пріоритетність ризику: так
- Згадки про інструменти безпеки, якщо є: Okta Support, HAR-файли, синхронізація Google-акаунта

### 3. Heroku - stolen GitHub OAuth tokens and config vars

- Рік і дата інциденту або публікації: квітень-травень 2022
- Тип джерела: офіційний розбір інциденту
- Джерела:
  - [Heroku April 2022 Incident Review](https://www.heroku.com/blog/april-2022-incident-review/)
- Короткий опис: Heroku повідомила, що зловмисник отримав доступ до бази з GitHub OAuth-токенами, завантажував приватні репозиторії та отримав конфігураційні змінні на рівні конвеєра доставки для Review Apps і Heroku CI.
- Використаний розрив у контролях або ризиковий розрив: Збережені OAuth-токени й конфігураційні змінні конвеєра доставки створили широкий радіус ураження між платформою як сервісом (PaaS), GitHub і CI/CD.
- Потенційний або підтверджений подальший рух атакувальника: Атакувальник переходив від компрометації машинного акаунта або токена до репозиторіїв GitHub і конфігурацій конвеєра доставки.
- Наслідки: Heroku відкликала токени, повідомляла клієнтів і ротувала доступи; потенційно постраждали приватні репозиторії й секрети.
- Релевантність до цієї загрози/ризику: Релевантність пряма: кейс підтверджує або ілюструє розрив у контролях або ризиковий розрив для загрози 'Немає єдиного джерела облікових записів' у зоні 'IdP / SSO / MFA', бо показує реальний наслідок схожого розриву: Збережені OAuth-токени й конфігураційні змінні конвеєра доставки створили широкий радіус ураження між платформою як сервісом (PaaS), GitHub і CI/CD.
- Рівень доказовості: високий
- Чи підсилює пріоритетність ризику: так
- Згадки про інструменти безпеки, якщо є: GitHub OAuth, Heroku CI, конфігураційні змінні

- Релевантність до цієї загрози/ризику: Релевантність пряма: кейс підтверджує або ілюструє розрив у контролях або ризиковий розрив для загрози 'Немає єдиного джерела облікових записів' у зоні 'IdP / SSO / MFA', бо показує реальний наслідок схожого розриву: Інструменти технічної підтримки або адміністрування залежали від доступів співробітників, які були вразливі до соціальної інженерії.

- Дата атаки або виявлення, якщо відома: виявлено 30 червня 2025
- Тип джерела: офіційне повідомлення компанії
- Джерела:
  - [Qantas cyber incident](https://www.qantasnewsroom.com.au/media-releases/qantas-cyber-incident/)
  - [Qantas update 9 July 2025](https://www.qantasnewsroom.com.au/media-releases/update-on-qantas-cyber-incident-wednesday-9-july-2025/)

- Використаний розрив у контролях або ризиковий розрив: Платформа підтримки або обслуговування клієнтів стороннього провайдера містила великий набір даних клієнтів і стала шляхом до персональних даних (персональні дані (PII)) без компрометації основних операційних процесів авіакомпанії.
- Потенційний або підтверджений подальший рух атакувальника: Qantas ізолювала та локалізувала скомпрометовану систему, працювала з кібербезпековими експертами, Australian Cyber Security Centre, регуляторами й поліцією.
- Наслідки: Було скомпрометовано імена клієнтів, адреси електронної пошти, номери телефонів, дати народження й дані програми лояльності; платіжні дані, паспортні дані, паролі й PIN-коди не були доступні зловмиснику за заявою Qantas.
- Релевантність до цієї загрози/ризику: Релевантність пряма: кейс ілюструє актуальний прояв такого розриву у 2025/2026 для загрози 'Немає єдиного джерела облікових записів' у зоні 'IdP / SSO / MFA', оскільки Платформа підтримки або обслуговування клієнтів стороннього провайдера містила великий набір даних клієнтів і стала шляхом до персональних даних (персональні дані (PII)) без компрометації основних операційних процесів авіакомпанії.
- Рівень доказовості: високий
- Чи підсилює пріоритетність ризику: так
- Згадки про інструменти безпеки, якщо є: платформа обслуговування клієнтів стороннього провайдера, контакт-центр, криміналістичний аналіз

### 2. Salesloft Drift - OAuth token theft і Salesforce data theft campaign

- Дата публічного розкриття, публікації або оновлення: серпень-вересень 2025
- Дата атаки або виявлення, якщо відома: серпень 2025; GitHub reconnaissance з березня 2025 за Mandiant/Salesloft reporting
- Тип джерела: звіт постачальника про загрозу / безпекова реакція платформи
- Джерела:
  - [Google Cloud / Mandiant](https://cloud.google.com/blog/topics/threat-intelligence/data-theft-salesforce-instances-via-salesloft-drift/)
  - [Salesforce security response](https://help.salesforce.com/s/articleView?id=005134951&language=en_US&type=1)
- Опис інциденту українською: Google Cloud/Mandiant описали кампанію викрадення даних, у якій зловмисник використовував скомпрометовані OAuth-токени, пов'язані зі стороннім застосунком Salesloft Drift, для доступу до клієнтських середовищ Salesforce.
- Використаний розрив у контролях або ризиковий розрив: Довірена SaaS/AI-інтеграція з довгоживучими OAuth-токени дала міжклієнтський радіус ураження для клієнтські CRM-дані.
- Потенційний або підтверджений подальший рух атакувальника: Зловмисник використовував OAuth-токени для доступу до середовищ Salesforce; Salesforce disabled Salesloft integrations і рекомендував переглянути, відкликати й ротувати токени.
- Наслідки: Публічні попередження описують викрадення даних із середовищ Salesforce і ризик розкриття контактних даних, звернень підтримки та даних, схожих на облікові, у CRM-записах.
- Релевантність до цієї загрози/ризику: Релевантність пряма: кейс ілюструє актуальний прояв такого розриву у 2025/2026 для загрози 'Немає єдиного джерела облікових записів' у зоні 'IdP / SSO / MFA', оскільки Довірена SaaS/AI-інтеграція з довгоживучими OAuth-токени дала міжклієнтський радіус ураження для клієнтські CRM-дані.
- Рівень доказовості: високий
- Чи підсилює пріоритетність ризику: так
- Згадки про інструменти безпеки, якщо є: Salesloft Drift, Salesforce, OAuth-токени, Google/Mandiant

### 3. ADT - unauthorized access до cloud-based environments

- Дата публічного розкриття, публікації або оновлення: 24 квітня 2026
- Дата атаки або виявлення, якщо відома: виявлено 20 квітня 2026
- Тип джерела: офіційна заява компанії
- Джерела:
  - [ADT newsroom](https://newsroom.adt.com/corporate-news/adt-detects-cybersecurity-incident)
- Опис інциденту українською: ADT повідомила, що системи кібербезпеки виявили несанкціонований доступ до обмежений набір даних клієнтів і потенційних клієнтів; компанія активувала протоколи реагування, припинила вторгнення, залучила сторонніх експертів і повідомила правоохоронні органи.
- Використаний розрив у контролях або ризиковий розрив: Хмарне середовище з даними клієнтів мав несанкціонований доступ path, який відкрив персональні дані (PII) без впливу на системи охоронної сигналізації та безпеки.
- Потенційний або підтверджений подальший рух атакувальника: ADT заявила про containment і відсутність компрометації систем безпеки клієнтів; доступ стосувався дані клієнтів і потенційних клієнтів.
- Наслідки: Розкрито імена, телефони, адреси, а в частини записів дати народження і останні чотири цифри номерів SSN/Tax ID; платіжні дані не зачеплено.
- Релевантність до цієї загрози/ризику: Релевантність пряма: кейс ілюструє актуальний прояв такого розриву у 2025/2026 для загрози 'Немає єдиного джерела облікових записів' у зоні 'IdP / SSO / MFA', оскільки Хмарне середовище з даними клієнтів мав несанкціонований доступ path, який відкрив персональні дані (PII) без впливу на системи охоронної сигналізації та безпеки.

## Висновок

- Повторювані патерни: Файли й артефакти технічної підтримки, HAR-файли й службові облікові дані не були достатньо ізольовані від персонального пристрою/акаунта. ; Доступ зони технічної підтримки/IdP залежав від окремого привілейованого робочого контуру та стороннього провайдера. ; Збережені OAuth-токени й конфігураційні змінні конвеєра доставки створили широкий радіус ураження між платформою як сервісом (PaaS), GitHub і CI/CD..
- Наскільки поширений тип проблеми: знайдені кейси показують, що цей клас розриву трапляється в реальних інцидентах із SaaS-сервісами, хмарними середовищами, CI/CD, пристроями або процесами управління безпекою; точна поширеність не оцінювалась кількісно.
- Чи варто залишити поточну пріоритетність: поточну пріоритетність `Висока` варто залишити, якщо внутрішній стан контролів відповідає описаному ризику; для непрямих кейсів це слід трактувати як підтримувальний, а не абсолютний доказ.
- Короткий текст для колонки "Пов'язані інциденти": [Okta / Sitel - компрометація support-доступу Lapsus$](access-idp-source-incident-research.md): Okta показала, що компрометація доступу технічної підтримки до IdP-зони може створити ризик для клієнтського контексту навіть без повного захоплення платформи для ризику `Немає єдиного джерела облікових записів`.
